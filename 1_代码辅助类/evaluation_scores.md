## 1. 评价指标说明（1–5 分）

| 指标 | 含义 | 5 分 | 1 分 |

## 7. 典型案例摘要

### 7.1 表现最佳

**Claude 3.5 Sonnet + 改进 Prompt + 样例 09（闰年）**  
完整给出 `(year % 400 == 0) or (year % 4 == 0 and year % 100 != 0)` 规则，六段式输出结构清晰，1900/2000/2024 验证说明完整。

### 7.2 表现较差

**Gemini 2.0 Flash + 基础 Prompt + 样例 09**  
仅保留 `% 4 == 0` 判断，1900 年误判为闰年；错误具有**隐蔽性**，初学者不易察觉。

### 7.3 Prompt 改进前后差异明显

**Qwen2.5-72B + 样例 06**  
- 基础：建议 `items[-1]` 等方案，未正确处理「取最后三个」语义（正确性 3）  
- 改进：明确使用 `items[-3:]`，并说明负索引切片（正确性 5）

### 7.4 过度重构反例（未出现严重情况）

各模型在改进 Prompt 约束下均较好遵守「最小改动」；Qwen 在样例 07 基础 Prompt 中曾建议完整 class 封装（未采纳进最终记录），改进 Prompt 后改为 `dict.get`。

## 8. 总体结论（供 LCL 汇总）

1. **代码调试任务上整体排序（改进 Prompt）**：Claude ≈ GPT-4o > DeepSeek > Qwen > Gemini，差距主要在逻辑错误与性能类样例。  
2. **改进 Prompt 价值**：结构化输出要求显著提升注释质量（+0.8~1.2 分）与鲁棒性（+0.4~0.6 分），对 Gemini/Qwen 的正确性提升最大。  
3. **LLM 局限**：逻辑语义类 bug（差一、业务规则）比语法/NameError 更易出错；性能优化类需模型具备算法意识。  
4. **人工复核建议**：逻辑类修复、涉及边界条件的代码、以及模型给出的「验证通过」声明，均需本地运行测试后再采纳。
